Vera vs Vow — agent-oriented language comparison

Two languages built for AI agents to write, not humans to read. Vera bets on removing variable names and Z3 with a runtime-fallback tier. Vow bets on BMC counterexamples, blame, and linear types. Where do they meet, and what is each worth learning from?
Compiled 2026-05-18 · Vera v0.0.155 (2026-05-13) · Vow self-hosted (current main) · branch vow/compareVera
Vera · aallan/vera · MIT · .vera Vow · self-hosted · .vow

TL;DR

Vera and Vow both claim the same tagline ("a language for AI agents") and they agree on the diagnosis: mandatory contracts, structured diagnostics, machine-first output. Then they diverge in almost every implementation choice. Vera drops variable names entirely — bindings are typed De Bruijn slots (@Int.0) — targets WebAssembly via a Python+Lark+Z3 toolchain, and discharges contracts with Z3 in three tiers (auto / hinted / runtime fallback). Vow keeps named variables, targets native via Cranelift, self-hosts to a binary fixed point, and discharges contracts with ESBMC bounded model checking that returns concrete counterexamples and blame. The two languages confirm the category is real but answer it very differently. Most directly portable from Vera: Inference as a tracked effect, contract-driven property tests, and fix examples per error code. Reject: De Bruijn names, WASM-only target, runtime-fallback verification tier.

3 · Syntax catalog (Vera)

Hello world — the surface shock

Vera · hello.vera

module hello;

public fn main(@Unit -> @Unit)
  requires(true)
  ensures(true)
  effects(<IO>)
{
  IO.print("Hello, world!\n")
}

Vow · hello.vow

module Hello

fn main() -> i32 [io] {
    print_str("Hello, world!");
    0
}

Even the trivial program is busier in Vera: public/private is mandatory, requires/ensures/effects are mandatory, the parameter is @Unit, the return is @Unit, and the call goes through the IO effect namespace.

De Bruijn slot references — the defining bet

Vera has no variable names. Every binding is a typed slot, referenced as @Type.N. Each type has its own counter; index 0 is the most recent binding of that type. Parameters bind left-to-right but the rightmost parameter of a given type is index 0.

private fn add(@Int, @Int -> @Int)
  requires(true)
  ensures(@Int.result == @Int.0 + @Int.1)
  effects(pure)
{ @Int.0 + @Int.1 }
-- @Int.0 = SECOND param (rightmost), @Int.1 = first param

private fn repeat(@String, @Int -> @String)
  requires(@Int.0 >= 0)
  ensures(true)
  effects(pure)
{ string_repeat(@String.0, @Int.0) }
-- Mixed types each have their own counter

-- let bindings push a new slot of that type, shifting older ones
let @Int = @Int.0 * 2;   -- new @Int.0 = (old @Int.0) * 2; old becomes @Int.1
let @Int = @Int.0 + 1;
@Int.0

-- Inside ensures, @T.result refers to the return value
-- Debug with: vera check --explain-slots file.vera

Functions, contracts, effects

private fn clamp(@Int, @Int, @Int -> @Int)
  requires(@Int.1 <= @Int.2)
  ensures(@Int.result >= @Int.1)
  ensures(@Int.result <= @Int.2)
  decreases(@Int.0)              -- termination measure (mandatory on pure recursion)
  effects(pure)
{
  if @Int.0 < @Int.1 then { @Int.1 }
  else { if @Int.0 > @Int.2 then { @Int.2 } else { @Int.0 } }
}

Multiple ensures lines conjoin. requires, ensures, and effects are mandatory on every function. decreases is mandatory on pure recursive functions. There is no invariant — Vera has no loops.

Effect handlers

-- Discharge a State effect with a pure handler
handle[State<Int>](@Int = 0) {
  get(@Unit) -> { resume(@Int.0) },
  put(@Int)  -> { resume(()) } with @Int = @Int.0
} in {
  put(42);
  get(())
}

-- Discharge an Exception effect
handle[Exn<String>] {
  throw(@String) -> None
} in { Some(safe_div(@Int.0, @Int.1)) }

ADTs, pattern matching, generics, abilities

private data List<T> { Nil, Cons(T, List<T>) }
private data Tree<T>  { Leaf, Node(Tree<T>, T, Tree<T>) }

-- Generic with ability (type-class) constraints
private forall<T where Eq<T>, Ord<T>> fn sort(@List<T>.0 -> @List<T>) ...

match @Option<Int>.0 {
  None        -> @Int.0,
  Some(@Int)  -> @Int.0       -- payload binds as a new @Int.0 in this arm
}

Built-in abilities: Eq, Ord, Hash, Show. Match is exhaustive. Patterns: constructors, literals, wildcard _.

Refinement types & typed holes

type PosInt = { @Int | @Int.0 > 0 };
type Name   = String;
type IntToInt = Fn(Int -> Int) effects(pure)

-- Typed holes (?) type-check but emit warning W001 and won't compile.
-- Lets the agent work incrementally.
private fn todo(@Int -> @Int)
  requires(true) ensures(true) effects(pure)
{ ? }

Quantified contracts

requires(forall(@Nat, array_length(@Array<Int>.0),
  fn(@Nat -> @Bool) effects(pure) { @Array<Int>.0[@Nat.0] > 0 }))

ensures(exists(@Nat, array_length(@Array<Int>.0),
  fn(@Nat -> @Bool) effects(pure) { @Array<Int>.0[@Nat.0] == @Int.result }))

Quantifiers are bounded over Nat ranges — not the unbounded ∀ of full SMT. Helps Z3 stay decidable.

Verification tiers (verbatim README)

  1. Tier 1 — decidable arithmetic, comparisons, Booleans, ADTs, termination. Discharged automatically by Z3.
  2. Tier 2 — guided. Hints supplied by the author when Z3 alone can't decide.
  3. Tier 3 — runtime check. The undecided postcondition becomes a runtime assert at codegen time.

Verify JSON includes tier1_verified and tier3_runtime counters. Tier 2 is described in the README but not directly exposed as a counter.

Error code ranges: W001 (typed holes) · E001–E007 parse · E010 transform · E120–E176 type-check · E200–E233 calls · E300–E335 control-flow/handlers · E500–E525 verification · E600–E614 codegen · E700–E702 testing. Each error ships a description + rationale + fix + spec_ref.

Not present

  • No LSP, no tree-sitter, no REPL. Editor support: a VSCode extension + TextMate bundle in editors/.
  • No project manifest for Vera code itself (no vera.toml). Vera is itself a Python package with a pyproject.toml.
  • No package registry / dependency manager for Vera. PyPI name vera is squatted — install only from GitHub source.
  • No source maps, stack traces, or debugger.
  • No build cache / artifact directory convention (_build/, dist/).

6 · AI angle

What Vera bets on at the AI layer

  • De Bruijn names — the headline bet. Hypothesis: removing names removes a hallucination class. Contested on HN.
  • Mandatory everything — public/private, requires, ensures, effects on every function. Less to forget, more boilerplate.
  • One canonical form — vera fmt --check means there is only one way to write any program.
  • Stable error codes with fix field — each diagnostic carries an error_code, a rationale, a fix suggestion, and a spec_ref. Marketed as "diagnostics as instructions."
  • Contract-driven property testing — vera test --trials N uses Z3 to synthesize inputs satisfying requires, runs them through WASM, checks ensures. No hand-written test cases needed.
  • Inference as a tracked effect — Inference.complete sits in the effect row. Pure functions cannot accidentally call an LLM.
  • In-tree agent docs — SKILL.md (Claude Code), AGENTS.md (Cursor/Windsurf/Copilot), CLAUDE.md, DE_BRUIJN.md, llms.txt. Auto-discovered.

50 problems × 6 models, 5 difficulty tiers (pure arithmetic → strings/arrays → ADTs → recursion+termination → multi-function with effects). Two modes: full-spec (problem ships contracts) and spec-from-NL (agent writes its own contracts).

The skill

Vera ships a single monolithic SKILL.md: 99 KB / 2,327 lines / ~24,900 tokens. Two sections account for ~45% of the file:

For scale: typical Claude Code skill guidance suggests an always-loaded SKILL.md under ~5,000 tokens, with reference material loaded on demand via linked sub-files. Vera is ~5× over that.

Vow ships a 1.7 KB / 48-line entrypoint SKILL.md that links to reference/grammar.md, reference/cli.md, etc.; the full self-contained bundle (~30 K tokens, comparable to Vera's) is printed on demand via vowc skill print --bundle rather than loaded by default.

The structural lesson on the skill. Vera treats the skill as "everything an agent needs in one fetch"; Vow treats it as an entrypoint that points at reference files. The Vera choice is defensible only if you assume the agent won't follow links — which, for some agents in production, is currently true. It is the wrong default for the Claude Code skill format specifically, where the body is always-loaded.

7 · Verification: Z3 vs ESBMC, head-to-head

PropertyVera (Z3)Vow (ESBMC)
What's being decidedValidity of first-order formulas over decidable theories (arithmetic, BV, arrays, ADTs).Reachability of assertion violations within a bounded unwinding of the program.

These are complementary, not strictly stronger/weaker. Z3 is better at pure arithmetic and quantified contracts. ESBMC is better at concrete bug-finding in imperative loops and memory access patterns. Vow gets witnesses; Vera gets quantifiers.

9 · What Vow can borrow from Vera

  1. Inference as a first-class effect.
    Vera puts LLM calls in the effect row: Inference.complete : String -> Result<String, String>, effects(<Inference>). A pure function can't accidentally call out to a remote model — it's a type error. If Vow ever grows agent-callable builtins (LLM-as-stdlib), this is the right shape. Cost: one effect kind, one runtime shim, no verifier impact. Strongest portable idea.
  2. Contract-driven property testing.
    vera test --trials N uses Z3 to synthesize inputs satisfying requires, then runs the function and asserts ensures. Functions taking ADT or function-type params are skipped with a message. Vow already has vowc test; adding a --from-contracts --trials N mode reuses the existing CEGIS infrastructure to generate random witnesses and exercise ensures at runtime. Useful belt-and-braces beside ESBMC.
  3. Each error code ships a fix example.
    Vera's diagnostics carry description, rationale, fix, and spec_ref. Vow's errors.md has rationale and description but less often a concrete wrong/right snippet. Documentation-only change; tightens the diagnostics-as-instructions story.
  4. Typed holes (?) as a warning, not a hard error.
    Vera lets you write { ? } as a function body. It type-checks (good shape, good name resolution) and emits W001, but won't compile. Lets an agent stub and iterate. Cost: parser + checker hook; small win for incremental development.
  5. Mandatory decreases on pure recursion (or some equivalent).
    Vera doesn't have loops, so it requires decreases on recursive pure functions. Vow uses bounded unwinding instead. But borrowing the idea — an explicit termination measure that the agent has to state — might let Vow drop ESBMC's unwind bound for some cases or sharpen counterexamples. Worth a feasibility note in the contracts ADR.
  6. One canonical form, enforced.
    Vera ships vera fmt --check as part of the workflow — there is no “style” debate. Vow already has a canonical printer (a compiler pass, not a formatter), but doesn't enforce it as part of vowc build. Optional CI gate.
  7. Bounded quantifiers in contracts.
    Vera's forall(@Nat, length, fn(...) {...}) is a quantifier over a known-bounded range. Translates straightforwardly to ESBMC as an unrolled loop of assertions. A small concrete win for contracts that today require N copy-pasted ensures lines.
  8. Spec chapter cross-references in diagnostics.
    Each Vera error carries spec_ref: "spec/05-contracts.md#postconditions". Vow's spec is already cleanly chaptered (docs/spec/{grammar,cli,contracts,errors,examples}.md); embedding the anchor in the diagnostic JSON closes the loop. The agent doesn't have to guess where to read.

10 · What Vow should not borrow

Vera's identity is built on choices that fight Vow's identity. Picking these up would dilute, not strengthen, what Vow is for.

  • De Bruijn slot references. Vera's defining bet, and the most contested by every external reviewer. The benchmark wins are model-specific (Kimi K2.5 100%, Claude Sonnet 4 79% — worse than Python). Names anchor LLM attention; grep matters; the cost to human collaborators is real even in an agent-first language. The thesis is too narrow and too disputed to bet on.
  • Tier-3 runtime fallback for verification. If Z3 can't discharge a contract, Vera silently emits a runtime assert and counts it in tier3_runtime. That breaks the "verified by default" pitch. Vow's "if ESBMC can't prove it, mark it as such and don't lie about the status" is the more honest position. Adopt the counting in JSON if you like — don't adopt the fallback.
  • WASM-only backend. Cranelift native + a binary fixed-point self-host is one of Vow's strongest credibility signals. Vera's Python-host + WASM-only target is a research-grade pipeline; do not chase it.
  • Mandatory effect & visibility on every function. Vera requires public/private, requires, ensures, and effects on every declaration. Even for trivial helpers. Vow's "pure-by-default, effect annotations only when needed" carries the same information with less syntactic load.
  • Monolithic 99 KB SKILL.md. Vow already does the right thing — small entrypoint, linked references, opt-in bundle. Vera's choice is defensible only for non-link-following agents; the always-loaded budget makes it the wrong default for Claude Code specifically.
  • No loops, no mutation, no linearity. Vera's "everything is recursion, everything copies, no ownership" model gives up Vow's linear-types story. Linear types are load-bearing for Vow's memory-discipline brand. Don't trade them.
  • Async-as-a-marker-only. Vera's Async effect is documented as having "no runtime overhead" because it's sequential. That's a UX trap — same syntax, different semantics, and an agent has no way to tell from the call site. If Vow ever adds async, it should be real.

11 · Bottom line

Vera is the second strong signal in two months (after MoonBit's 0.9 verification release) that "languages for AI agents with mandatory contracts" is a real category, not a fad. Both projects converged on the same diagnosis: machine-first error output, contracts as the source of truth, agent-facing docs in-tree. They diverge on three big bets:

  1. Names: Vera removes them. Vow keeps them. The external evidence so far favors Vow.
  2. Verifier: Vera uses Z3 with a runtime-fallback tier. Vow uses ESBMC with concrete counterexamples and blame. Complementary, not strictly ordered.
  3. Implementation maturity: Vera is a 3-month-old solo project with a Python reference compiler and ~350 stars. Vow is self-hosted to a binary fixed point with a published benchmark suite. The maturity gap matters more than the design gap.

Vow's moat is the verifier, blame, linear types, and self-hosting. Vera doesn't threaten any of those. The most actionable items from this comparison, in priority order:

  1. Add Inference as a tracked effect — clean win when Vow grows LLM-callable builtins.
  2. Add a --from-contracts mode to vowc test — reuses CEGIS infrastructure for property tests.
  3. Add a fix: wrong/right snippet to every E### in errors.md — documentation-only, no code changes.
  4. Embed spec_ref anchors in diagnostic JSON — tightens the agent loop.
  5. Consider bounded quantifiers in contracts — small win, ESBMC-compatible via unrolling.

Reject De Bruijn names, the runtime-fallback tier, WASM-only target, and the monolithic skill. Those are Vera's identity, not Vow's.
