Zero vs Vow — agent-oriented language comparison

Two languages built for AI agents. One bets on structured diagnostics. The other bets on formal verification. Where do they meet, and what does each have to teach the other?
Compiled May 18, 2026 · Zero v0.1.2 (released May 15, 2026) · Vow self-hosted (current main)
Zero · vercel-labs/zero · Apache-2.0 · .0 Vow · self-hosted · .vow

TL;DR

Zero and Vow agree on the diagnosis: error messages should be machine-readable first, humans second. They diverge sharply on the cure. Zero ships a richly structured JSON surface across every CLI subcommand and a fixSafety taxonomy on each diagnostic — but has no formal verifier, no contracts, and no theorem prover. Vow ships ESBMC bounded model checking with requires / ensures / invariant contracts, blame semantics, and a CEGIS loop — but has less variety in agent-facing JSON surfaces. The two designs are complementary, not competitive. The most directly portable ideas Vow can borrow from Zero are: the discovery-stub + binary-pinned skill bundle (zero skills get), the fixSafety ladder on every diagnostic, and typed repair handles that name a fix rather than just describing the failure.

choice ParseResult {
    ok: i32,
    err: String,
}

fun validate(ok: Bool) -> i32 raises { InvalidInput } {
    if ok == false { raise InvalidInput }
    return 42
}

fun classify(raw: Bool) -> ParseResult {
    let value = validate(raw) rescue err { return ParseResult.err("bad input") }
    return ParseResult.ok(value)
}

pub fun main(world: World) -> Void raises {
    let result = classify(true)
    match result {
        .ok => value {
            if value == 42 { check world.out.write("validated\n") }
        }
        .err => message {
            check world.err.write("failed\n")
        }
    }
}

Diagnostic JSON shape

{
  "schemaVersion": 1,
  "ok": false,
  "diagnostics": [
    {
      "severity": "error",
      "code": "NAM003",
      "message": "...",
      "path": "src/main.0",
      "line": 12, "column": 5, "length": 7,
      "expected": "...", "actual": "...",
      "help": "...",
      "fixSafety": "local-edit",
      "repair": { "id": "fix-import-path", "summary": "..." },
      "related": [ ... ]
    }
  ]
}

fixSafety classifies the repair, not the violation, on a 5-level ladder:

  • format-only — whitespace, formatting
  • behavior-preserving — refactor with no semantic change
  • local-edit — localized semantic change
  • api-changing — changes public surface
  • requires-human-review — "planning hint, not an automatic patch"

5 · The skill: Zero vs Vow

Both projects ship a Claude Code skill. They make radically different structural choices.

The big structural lesson. Zero treats the skill itself as a thing the compiler emits, just like diagnostics or build output. The on-disk SKILL.md is a 1.5 KB pointer; the agent asks the running compiler what its skill is, and the answer matches that compiler's exact version. Vow has the same affordance (vowc skill print) but ships the full content on disk rather than indexing into the binary. Zero's choice is more aggressive about closing the version-skew loophole, at the cost of needing a working binary before you can read the skill.

6 · What Vow can borrow from Zero

  1. Add a fixSafety-style ladder on every diagnostic.
    Vow already has Blame (who is at fault) but no notion of how dangerous the fix is. A 5-level ladder — format-only, behavior-preserving, local-edit, api-changing, requires-human-review — would let an agent triage automatically. For Vow specifically: a violation that suggests strengthening a requires clause is api-changing; one that suggests adding a loop invariant is local-edit; a counterexample showing a genuine logic bug is requires-human-review.
  2. Emit a typed repair handle alongside the diagnostic.
    Today a vow violation says "requires: y != 0 failed with y == 0". Adding repair: { id: "guard-divisor", summary: "Add precondition or rescue" } turns the diagnostic into an actionable JSON object the agent can dispatch on. The compiler doesn't have to write the patch — just name it. Zero's zero-diagnostics shows the shape.
  3. Discovery-stub pattern for the on-disk skill.
    Vow already ships a stub SKILL.md with progressive disclosure. Going further — making it truly a discovery stub that says "ask the binary" — would close the version-skew window. Replace inline reference content with !`vowc skill get grammar --json`-style runtime queries. Practical hybrid: keep the on-disk reference for offline use, but stamp it with the compiler version it was generated from and have the skill tell the agent to regenerate if versions diverge.
  4. Split the skill into named sub-skills.
    Zero's seven micro-skills (zero-agent, zero-language, zero-diagnostics, zero-builds, zero-packages, zero-stdlib, zero-testing) average 2–3 KB each. An agent loads only what it needs. Vow's reference files are already structured this way (grammar.md, cli.md, contracts.md, errors.md, examples.md) — the missing piece is a discoverable index command: vowc skill list / vowc skill get contracts.
  5. Add vowc explain <error_code> as a first-class subcommand.
    Zero's zero explain NAM003 --json returns { code, title, fixSafety, defaultOutput }. For Vow, vowc explain VowViolation --json could return the violation taxonomy, blame model, and a pointer to contracts.md. Today this content lives only in errors.md; lifting it into a runtime command makes it discoverable from the JSON itself.
  6. An explicit "Do nots" page in the skill.
    Zero's zero-agent opens with six imperatives: don't invent syntax, don't invent CLI fields, don't scrape prose when JSON is available, treat effects as capabilities, prefer narrow commands, don't auto-apply requires-human-review. Vow's equivalents are spread across contracts.md and CLAUDE.md. Centralizing them into a short, list-of-don'ts page (or top of SKILL.md) would make them harder to miss. Vow-specific candidates: "Don't weaken contracts to appease ESBMC", "Don't add n <= 10-style verifier bounds to requires", "Don't use --no-verify to skip a real failure".
  7. Two-phase fix: plan, then apply.
    zero fix --plan --json proposes; the agent applies. Vow could add vowc fix --plan --json <file.vow> that, given a vow violation, emits a structured proposal — not a patch — covering candidate strengthenings of contracts, loop invariants, or guards. The agent decides whether to apply. The plan can be regression-tested independently of any patcher.
  8. Capability-passing as a UX layer over Vow's effect sets.
    Vow's effect sets are type-checked but invisible at the call site. Zero forces world: World through every effectful function, which is agent-greppable. This is probably too verbose for Vow's taste — but it's worth considering whether some effects (e.g. unbounded allocation, network) should be passed explicitly. The reverse question is also worth asking: should Zero adopt Vow-style implicit effect sets to reduce visual noise? Neither answer is obvious.
  9. Surface every cache/key/fingerprint that affects determinism.
    Zero exposes cacheKeyInputs, snapshotKey, releaseTargetContract, interfaceFingerprints. Vow already has a deterministic compile-object cache with a content-addressed key (see cli.md); making that key visible in the JSON build output gives agents a stable handle for cache diagnostics, regression detection, and reproducibility audits.
  10. Inline test "name" { expect(...) } blocks.
    Zero's inline test blocks (any .0 file can carry tests) lower the friction of test-writing for agents. Vow has tests/run/*.vow directory-based tests; inline tests would be a complementary mode for unit-level coverage. The unexpectedPasses idea — flagging when an xfail: starts passing — is independently useful as a regression-direction signal.

7 · What Vow should not borrow

Zero's correctness story is the shallow-end version of Vow's. "Types + capabilities + structured diagnostics" is real value but it's not formal verification. Vow shouldn't dilute its CEGIS / ESBMC story to chase Zero's UX. The two design positions are complementary, not competing:

  • Generics & meta(...). Zero has both. Vow has deliberately rejected both because they expand the verification surface. The "crisp rule" in CLAUDE.md — "add surface sugar only when it desugars to today's core semantics with near-zero verifier impact; reject anything that introduces a new type-system axis" — argues against this directly. Generics introduce monomorphization complexity that ESBMC has to chase.
  • Sub-10 KiB binary chase. Vow's binaries can be larger if it buys verification. Don't optimize away ESBMC's static C model for binary-size points.
  • Removing the C backend. Zero removed its generated-C backend in favor of direct emitters. Vow needs the C backend for ESBMC. Different priorities.
  • 25 subcommands. Zero's surface area is impressive but a lot of it (zero mem, zero ship, zero abi, zero doctor) is genuinely useful only at the scale of a productized toolchain. Vow should pick subcommands by clear agent need, not by parity.
  • Effect-as-capability syntax. Threading world: World through every effectful function is grep-friendly but visually heavy. Vow's effect sets on function types do the same job structurally with less noise. Adopt the explicitness; keep the syntax.

8 · Bottom line

Zero is the strongest signal yet that "languages designed for AI agents" is a real category, not a fad. It validates Vow's bet that machine-readable, structured compiler output is the right default. But it leaves the formal verification ground entirely uncontested. There is no other language in the agent-first niche that ships a bounded model checker, contracts with blame, and a CEGIS loop. Vow's moat is the verifier and the published benchmarks.

The most actionable items, in priority order:

  1. Add fixSafety on every diagnostic.
  2. Add typed repair handles in diagnostic JSON.
  3. Add vowc explain <code> and vowc skill list / get.
  4. Centralize a "Do nots" page in the skill.
  5. Two-phase vowc fix --plan --json.
