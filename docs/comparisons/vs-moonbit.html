MoonBit vs Vow

A comparative report on two languages pitched at AI-agent code generation.
Compiled 2026-05-18 · branch vow/compareMoonbit · MoonBit at 0.9 (Apr 2026)

TL;DR

MoonBit and Vow share a tagline ("a language for AI agents") and very little else. MoonBit is a broad, batteries-included WebAssembly-first language whose AI bet is at the sampling layer (constrained decoding) plus a vertically integrated agent (MoonBit Pilot). Verification, added in 0.9 (April 2026), is a Why3 + SMT deductive layer, opt-in per package, with separate .mbtp predicate files.

Vow is a deliberately small language whose AI bet is at the verification layer: inline requires/ensures/invariant verified by ESBMC, structured JSON counterexamples, blame attribution, linear types, and a CEGIS loop. No first-party agent, no closed compiler, no proprietary protocol.

They are not direct competitors. MoonBit confirms that the market for "AI-friendly languages with provable contracts" is real; Vow has a different and arguably tighter answer for the repair loop.

2. "For AI agents" — what each one means

MoonBit attacks the model layer

  • Constrained decoding: a "local sampler" rejects syntactically invalid tokens during generation; a "global sampler" type-checks. Hosted at ai.moonbitlang.com.
  • Pilot: first-party multi-modal coding agent (CLI + IDE + cloud), BYO model key.
  • Custom Agent Server Protocol (ASP) for cloud agent orchestration (not MCP).
  • Syntax research: top-level type signatures mandatory, "no nesting" preference, "KV-cache friendly" structure. (LLM4Code 2024 paper.)

Vow attacks the verification layer

  • Inline contracts: requires / ensures / invariant in the same expression language as code.
  • ESBMC bounded model checking: returns concrete counterexample traces, not just "unknown."
  • Blame semantics: requires blames Caller; ensures/invariant blames Callee.
  • Structured JSON diagnostics as a first-class agent contract — no proprietary protocol layered on top.
  • Intentionally small surface: no generics, traits, closures, macros, async.

Both projects converged on "verification matters" — MoonBit's 0.9 in April 2026 is direct evidence. They got there via different mechanisms (SMT/Why3 vs. BMC/ESBMC), with different ergonomics (separate predicate files vs. inline), and different counterexample shapes (per-VC verdict vs. concrete trace).

3. Syntax reference

MoonBit's surface is Rust-flavoured ML: fn, let, match, enum, struct, expression-oriented bodies. Examples below are verbatim from the official docs and the moonbit-community/verified repo.

3.1 Variables & mutability

let a = true
let b = false
let c = a && b

let mut i = 5
while i > 0 {
  println(i)
  i = i - 1
}

let r : Ref[Int] = { val: 100 }
r.val = 200
r.val += 1

Ref[T] is a single-field mut struct — needed because closures cannot capture let mut bindings.

3.2 Functions

fn add3(x : Int, y : Int, z : Int) -> Int {
  x + y + z
}

fn local_1() -> Int {
  fn inc(x) { x + 1 }                // local fn: inferred
  (fn(x) { x + inc(2) })(6)          // anonymous fn
}

[1, 2, 3].each(x => println(x * x))  // arrow lambda

fn labelled(arg1~ : Int, arg2~ : Int) -> Int { arg1 + arg2 }
let _ = labelled(arg2=2, arg1=1)

Top-level signatures are mandatory; locals are inferred. arg~ declares a labelled (keyword) argument.

3.3 Structs

struct User {
  id : Int
  name : String
  mut email : String
}

let u = User::{ id: 0, name: "John Doe", email: "john@doe.com" }
u.email = "john@doe.name"

let updated = { ..u, email: "john@doe.name" }  // functional record update

3.4 Enums & pattern matching

enum Lst {
  Nil
  Cons(Int, Lst)
}

fn sum(l : Lst) -> Int {
  match l {
    Nil => 0
    Cons(x, xs) => x + sum(xs)
  }
}

3.5 Generics & traits

fn[S, T] List::map(self : List[S], f : (S) -> T) -> List[T] {
  match self {
    Nil => Nil
    Cons(x, xs) => Cons(f(x), xs.map(f))
  }
}

pub(open) trait MyShow {
  to_string(Self) -> String
}

struct MyType {}
pub impl MyShow for MyType with to_string(self) { "MyType{}" }

struct T { a : Int; b : Int } derive(Eq, Compare, Debug, Default)

Generic parameters come before the function name (fn[T, U]), which is unusual among ML-family languages.

3.6 Pattern exotics

match ary {
  [] => "empty"
  [a, b, .. rest] => "head two"
  [.., a, b] => "tail two"
}

match c {
  'a'..='z' => "lower"
  'A'..='Z' => "upper"
  _         => "other"
}

match map {
  { "b"? : None, "a": x, .. } => x   // "b" must be absent
}

match opt {
  Some(a) if f(a) is [0, b] => a + b // guard + is-pattern
  Some(b) => b
  None    => -1
}

3.7 Error effects

suberror DivError { DivError(String) } derive(Debug)

fn div(x : Int, y : Int) -> Int raise DivError {
  if y == 0 { raise DivError("division by zero") }
  x / y
}

fn main {
  try div(42, 0) catch {
    DivError(s) => println(s)
  } noraise {
    v => println(v)
  }
  let res = try? (div(6, 0) * div(6, 3))   // ? wraps into Result
  let v   = try! div(6, 3)                 // ! panics on error
}

Errors are an effect on the return type (-> Int raise DivError) — checked, with no Result plumbing at call sites.

3.8 Async

async fn fetch_homepage() -> String {
  let (response, body) = @http.get("https://www.moonbitlang.com")
  guard response.code is (200..<300) else {
    fail("server responded with \{response.code} \{response.reason}")
  }
  body.text()
}

async fn main {
  @async.with_task_group(group => {
    group.spawn_bg(() => { let _ = fetch_homepage() })
  })
}

No await keyword — every call into an async fn is implicitly a suspend point. Structured concurrency via with_task_group.

3.10 Inline tests

test "addition" {
  assert_eq(1 + 1, 2)
}

test "snapshot" {
  inspect({ x: 10 }, content="{ x: 10 }")  // --update auto-rewrites content=
}

Snapshot strings live inline in source and are rewritten in place by the test runner — no separate golden directory.

5. Tooling

5.1 The moon CLI

Single CLI surface. Subcommands:

  • new, build, check, run, test, bench, clean, fmt, doc, info
  • add, remove, install, tree, update — dependency management
  • login, register, publish, package — registry operations
  • coverage analyze | report | clean
  • prove — new in 0.9 for formal verification (peer of build/test)
  • upgrade, shell-completion, generate-build-matrix, version

5.2 Files & manifests

  • .mbt — source files.
  • .mbti — generated interface file (moon info).
  • .mbtx — runnable scripts.
  • .mbtp — predicate file (0.9; appears in community verified-examples repo, not yet in the official file-extension reference).
  • moon.mod.json — module manifest (semver, deps, license, supported-targets, scripts).
  • moon.pkg.json — per-package config (imports, link, native-stub, formatter ignore, "proof-enabled": true for 0.9 verification).

5.3 Editor & runner

  • LSP: separate binary installed by the MoonBit toolchain; flagship VSCode extension moonbit.moonbit-lang.
  • Formatter: moon fmt.
  • Tests: inline test "name" { … } blocks discovered automatically; parallel run; snapshot tests rewrite source in place.
  • No REPL; closest is the web tour at tour.moonbitlang.com.
  • No first-party debugger; coverage instrumentation via moon coverage.

6. AI tooling

6.1 MoonBit Pilot

A multi-modal coding agent shipped with the toolchain:

  • Three surfaces: CLI (moon pilot), IDE integration in VSCode, and a cloud UI for async agent execution.
  • BYO model API key — documented for Claude Sonnet 4, Kimi K2, DeepSeek V3, Gemini 2.5 Pro. Per-token costs go to the provider.
  • Local config under ~/.moonagent/ and per-project .moonagent/moonagent.yml.
  • No public price list. Practically free-to-try, with no commitment that it remains so.

6.2 Agent Server Protocol (ASP)

MoonBit-custom protocol announced alongside Pilot. No public spec page, no schema, no comparison-to-MCP document. Functionally it sits at a different layer than MCP — ASP is about orchestrating multiple agent tasks, MCP is about exposing tools/resources to a single agent.

No first-party MCP integration. Pilot's documented configuration has no MCP server entries. A community library gmlewis/moonbit-mcp exists for writing MCP servers in MoonBit, but there's nothing in the toolchain for consuming third-party MCP servers from Pilot.

6.3 Constrained decoding

The most distinctive AI bet. Documented in the LLM4Code 2024 paper and hosted at ai.moonbitlang.com:

  • Local sampling — real-time syntactic adjustment of generated tokens against the MoonBit grammar.
  • Global sampling — type-checking accepted token spans; rejected spans backtrack with error fed to the model.
  • Reported large compilation-rate improvement at ~3% latency penalty.
  • Exploits MoonBit's "interface-before-implementation" linear program order to keep KV-cache misses low.
  • Not packaged as a standalone library — runs inside MoonBit's hosted AI service.

6.4 No "skill" for MoonBit

Despite the agent positioning, MoonBit does not publish a Claude Code / Cursor skill, agent profile, or system prompt. The agent angle is served by their own Pilot stack and ASP, not by integrating with the broader agent-tool ecosystem. This is a meaningful contrast with Vow, which ships a skill (build/vowc skill install) and depends on structured-JSON-everywhere as its agent contract.

7. Verification (MoonBit 0.9, April 2026)

First-class formal verification keywords: proof_require, proof_ensure, proof_invariant, proof_assert, proof_yield, proof_decrease, proof_reasoning, and #proof_pure. Predicates live in sibling .mbtp files. Backend is Why3 dispatching to a portfolio of SMT solvers (Z3, Alt-Ergo, CVC5). Opt-in per package via "proof-enabled": true in moon.pkg. Driven by moon prove.

7.1 Smallest example — abs

abs.mbt:

pub fn abs(x : Int) -> Int where { proof_ensure: result => abs_ok(x, result) } {
  if x >= 0 { x } else { -x }
}

abs.mbtp:

predicate abs_ok(x : Int, result : Int) {
  (result >= 0) && ((result == x) || (result == -x))
}

7.2 Binary search — function contract + loop invariants

pub fn[T : Compare] binary_search(
  xs : FixedArray[T], key : T,
) -> Int? where {
  proof_require: sorted(xs),
  proof_ensure: result => binary_search_ok(xs, key, result),
} {
  for i = 0, j = xs.length(); i < j; {
    let h = i + (j - i) / 2
    if xs[h] < key {
      proof_assert ∀ idx : Int, 0 <= idx && idx < h + 1 → xs[idx] < key
      continue h + 1, j
    } else if key < xs[h] {
      proof_assert ∀ idx : Int, h <= idx && idx < xs.length() → key < xs[idx]
      continue i, h
    } else {
      proof_assert xs[h] == key
      break Some(h)
    }
  } nobreak { None } where {
    proof_invariant: 0 <= i,
    proof_invariant: i <= j,
    proof_invariant: j <= xs.length(),
    proof_invariant: ∀ idx : Int, 0 <= idx && idx < i → xs[idx] < key,
    proof_invariant: ∀ idx : Int, j <= idx && idx < xs.length() → key < xs[idx],
    proof_yield: res => binary_search_ok(xs, key, res),
  }
}

7.3 The predicate language

predicate[T] in_bounds(xs : FixedArray[T], i : Int) {
  (0 <= i) && (i < xs.length())
}
predicate[T : Compare] sorted(xs : FixedArray[T]) {
  ∀ i : Int, ∀ j : Int,
    ((in_bounds(xs, i)) && (in_bounds(xs, j)) && (i <= j)) → xs[i] <= xs[j]
}
predicate[T : Compare] binary_search_ok(
  xs : FixedArray[T], key : T, result : Option[Int],
) {
  match result {
    None         => ∀ i : Int, in_bounds(xs, i) → xs[i] != key
    Some(result) => (in_bounds(xs, result)) && xs[result] == key
  }
}

Features of the predicate sublanguage: universal quantification via Unicode ∀; implication via →; recursive predicates over ADTs via match; generic predicates with the same trait-bound syntax as functions; finite-set/sequence/map model libraries in libs/; explicit proof_decrease measures for termination of recursive predicates.

7.4 Counterexample shape

Per-VC verdict from Why3 ("could not prove goal X" / Unknown / Timeout). No published JSON schema, no concrete program-trace counterexample. Reports land in _build/verif/. Vow's ESBMC pipeline, by contrast, returns concrete failing inputs and execution traces — the difference between deductive verification and bounded model checking.

7.5 AI integration

The framing is "AI guesses, prover checks." Pilot drafts predicates, invariants, and proof_assert hints; moon prove is the oracle. The release notes credit AI for generating the binary-search and AVL invariants. There is no language-level CEGIS protocol; the agent loops on whatever the prover says it couldn't prove.

7.6 Scope

Opt-in per package, per function. Stdlib moonbitlang/core is not verified end-to-end; a parallel ecosystem moonbit-community/verified hosts proven examples (binary search, AVL trees, heaps, plus finance case studies: stablecoin engine, AMM, lending, batch auction).

8.1 Recurring critiques

  • Licensing isn't really open. The Dec 2024 "open-source" release used the MoonBit Public Source License — an SSPL-style anti-cloud clause that critics on HN flagged as source-available, not OSI-open.
  • Benchmark concerns. A Lobsters thread documented a handicapped Rust FFT baseline used in a "30% faster than Rust" social-media campaign; corrective PRs reportedly went unaddressed.
  • Governance. Project led by IDEA (Shenzhen). Generic "do your own due diligence" caution in commentary; no specific export-control incidents found.

10. What Vow can learn (and what to reject)

  1. Constrained decoding is the layer above Vow. MoonBit attacks "agents emit invalid code" at the sampler. Vow attacks the same problem at the verifier. These compose: a Vow-aware sampler could rule out unparseable or trivially ill-typed tokens before ESBMC ever runs. Worth a forward-looking note in docs/spec/.
  2. Predicate files are tempting but expensive. Splitting predicates into .mbtp gives reuse and a richer logical sublanguage (quantifiers, recursive predicates) — but it's a new namespace, a new file type, and a new logic that humans and agents have to learn separately from the rest of the language. Fails the "doesn't make verification harder" gate unless there's a concrete reuse story Vow can't currently express.
  3. Why3 is a real escape hatch. If Vow ever wants ∀/∃ over unbounded structures, dispatching to Why3 (rather than reimplementing) is the boring choice. But it means giving up concrete counterexamples — the CEGIS loop changes shape.
  4. Inline snapshot tests are good ergonomics. MoonBit's inspect({ x: 10 }, content="...") auto-rewriting on --update is a tight loop worth borrowing for Vow's tests/run/ golden-output cases.
  5. Reject the proprietary protocol. ASP is closed, undocumented, and ships with the compiler. Vow's bet on "JSON-everywhere, no proprietary protocol" is the better story if the agent ecosystem is going to be Cambrian.
  6. Reject closed-source. MoonBit's source-available license is a recurring friction point in HN discussion. Vow's open-source posture is straightforward marketing as well as engineering.
  7. The big confirmation. MoonBit reaching for formal verification in 0.9 (April 2026) is direct market evidence that "AI-friendly language with provable contracts" is the right destination. Vow got there first via a different (arguably tighter for repair loops) route — BMC + concrete counterexamples + blame. Worth saying so loudly in any external-facing Vow material.
