Agent-oriented languages — what Vow can learn

Unified synthesis of three comparisons: MoonBit, Vera, and Zero against Vow. The category is real. The bets diverge sharply. A handful of ideas are worth porting; most are identity choices that don't belong in Vow.
Compiled 2026-05-19 · consolidates moonbit-comparison, vera-vs-vow, vera-skill-report, and zero-vs-vow.
MoonBit · v0.9 · Apr 2026 Vera · v0.0.155 · May 2026 Zero · v0.1.2 · May 2026 Vow · self-hosted main

TL;DR

Four languages now pitch themselves at AI agents. They agree on the diagnosis — machine-first compiler output, contracts as source of truth, in-tree agent docs, stable error codes — and confirm the category. They diverge on the cure:

  • MoonBit attacks the sampling layer (constrained decoding) plus a vertically integrated agent (Pilot, ASP). Verification arrived in 0.9 via Why3+SMT, opt-in per package, with separate .mbtp predicate files.
  • Vera attacks the name layer: typed De Bruijn slots (@Int.0) replace identifiers, plus mandatory contracts verified by Z3 in three tiers ending in runtime fallback.
  • Zero attacks the diagnostic layer: ~25 JSON-first subcommands, a fixSafety taxonomy on every repair, capability-passing for effects. No verifier, no contracts.
  • Vow attacks the verification layer: inline contracts, ESBMC bounded model checking, concrete counterexamples, Caller/Callee blame, linear types, binary-fixed-point self-host.

The action list for Vow: borrow Inference as a tracked effect (Vera), a fixSafety ladder + typed repair handles (Zero), fix examples per error code + spec_ref anchors (Vera), contract-driven property tests (Vera), bounded quantifiers (Vera), and two-phase fix --plan --json + explain <code> (Zero). Reject De Bruijn names, the runtime-fallback tier, generics+metaprogramming, monolithic SKILL.md, and Pilot-style proprietary protocols. Vow's moat is the verifier, blame, linear types, and self-hosting — none of the three threaten it.

1 · The landscape at a glance

MoonBit

Identity

IDEA (Shenzhen). Lead: Hongbo Zhang (ex-ReScript). Started Oct 2022; v0.9 April 2026. Source-available (MPSL, SSPL-style). Compiler in OCaml, distributed as WASM.

Bet

Constrained decoding at the sampler + first-party Pilot agent + custom ASP protocol + (new) Why3-backed deductive verification.

Numbers

~1.1k stars on core, ~2.3k on docs. Discord ~578. No named production users. LLM4Code 2024 paper.

Vera

Identity

Alasdair Allan (Negroni Venture Studios), solo. Public April 2026. MIT. Python+Lark+Z3+wasmtime, ~10 KLOC. WASM-only target.

Bet

De Bruijn slot references replace variable names. Mandatory requires/ensures/effects. Inference as a first-class effect.

Numbers

~351 stars, 17 forks, 156 releases in ~3 months. One HN thread (95 comments, mostly skeptical). VeraBench: 50 problems × 6 models.

Zero

Identity

Vercel Labs (Chris Tate, Matt Van Horn). Released v0.1.0 May 15, 2026. Apache-2.0. C compiler with direct native emitters (ELF / Mach-O / COFF / WASM). No LLVM.

Bet

Structured-JSON everywhere (~25 CLI subcommands). fixSafety ladder on every diagnostic. Capability-passing for effects (world: World).

Numbers

~1.9k stars in days. No published benchmarks. Sub-10 KiB native binaries. Self-host in progress.

Vow

Identity

Paulo Matos & collaborators. Open source. Cranelift native via FFI shim + ESBMC pipeline in parallel via C emitter. Binary fixed-point self-host.

Bet

Inline contracts (requires/ensures/invariant) verified by ESBMC BMC. Concrete counterexamples + Caller/Callee blame + CEGIS loop. Linear types. JSON-everywhere.

Numbers

36/36 Vow suite (sonnet-4-6); HumanEval 66/67 (sonnet-4); 103/103 non-stretch references verified by self-hosted compiler.

2 · What everyone agrees on

Four projects, four bets, but a single converged thesis. Every one of them ships:

  • Machine-readable error output as the primary surface, with human-readable as the alternate. Stable diagnostic codes (E###, NAM003, VowViolation) with description + rationale + fix + spec_ref-style fields.
  • Effects in the type system. All four track side effects as part of the function type (or equivalent capability handle). Pure functions cannot accidentally do IO.
  • Agent-facing docs in-tree. A SKILL.md (MoonBit is the lone holdout), AGENTS.md, CLAUDE.md, or compiler-served equivalent. Auto-discoverable, not buried in a wiki.
  • Contracts as the source of truth (3 of 4 — Zero is the exception): MoonBit's proof_require/proof_ensure, Vera's mandatory requires/ensures, Vow's vow blocks. The disagreement is over the verifier, not the idea.
  • JSON build artifacts. Diagnostic JSON, build JSON, verification JSON. Terminal output is for humans; agents consume the schema.

Four projects converging on the same diagnosis within a 12-month window (MoonBit 0.9 Apr 2026, Vera Apr 2026, Zero May 2026, Vow self-host before that) is the strongest evidence yet that "language designed for agents with mandatory contracts and machine-first output" is a real category. The market is now contested. Vow's positioning needs to be sharper than "we also do this".

3 · The four bets

Where they diverge is more interesting than where they agree. Each project picked exactly one layer of the agentic-coding stack to attack:

Layer attackedProjectMechanismWhat it gives up
Sampling MoonBit Constrained decoding (local syntactic + global type-checking samplers) running on hosted infra; ~3% latency penalty for large compile-rate wins on small models. Coupling to hosted infra; closed sampler; not packaged as standalone lib.
Naming Vera Eliminate variable names entirely. Typed De Bruijn slots (@Int.0) with per-type counters. Grep-ability; local-reasoning when editing top of function; cross-model bench wins are mixed (Kimi K2.5 100%, Sonnet 4 79%).
Diagnostics Zero fixSafety ladder + typed repair handles + ~25 JSON-first subcommands + binary-pinned skills. No verifier; no formal correctness story; types + capabilities only.
Verification Vow Inline contracts → ESBMC BMC → concrete counterexample with values + trace + blame → agent fixes → rerun. CEGIS loop is the headline. No quantifiers in contracts; bounded by unwind depth; verifier latency.

These are complementary, not competing. A serious agent-language toolchain could (in principle) compose all four: a constrained sampler emits tokens that satisfy the grammar; a small core surface keeps the verifier tractable; structured diagnostics drive the repair loop; concrete counterexamples close it.

4 · Verification head-to-head

The most consequential axis of comparison. Three of the four (MoonBit, Vera, Vow) ship a verifier; Zero deliberately doesn't.

Property MoonBit 0.9 Vera Zero Vow
Backend Why3 → Z3/Alt-Ergo/CVC5 Z3 SMT directly None ESBMC (BMC)
What's being decided Validity of FO formulas in decidable theories. Same, but no separate predicate logic. N/A — types + capabilities only. Reachability of assertion violation in bounded unwind.
Predicate location Separate .mbtp file w/ dedicated logical sublanguage. Inline. N/A Inline, same expression language as code.
Quantifiers ∀, recursive predicates, finite-set/seq/map model libs. Bounded forall/exists over Nat ranges. N/A None — boolean exprs only.
Failure shape Per-VC verdict ("could not prove goal X" / Unknown / Timeout). Per-VC verdict; if undecided, becomes Tier-3 runtime assert (E500–E525). Type/capability error JSON. Concrete failing input + execution trace + JSON + Caller/Callee blame.
Termination proof_decrease measures. Mandatory decreases on pure recursion (no loops in language). N/A Bounded unwinding instead.
Loops Yes — proof_invariant clauses. None — rejected at language level. Yes — no formal reasoning. Yes — invariant + bounded unwind.
Blame None. None. N/A; fixSafety classifies repairs instead. First-class Caller/Callee in JSON.
Trigger Opt-in per package ("proof-enabled": true); moon prove. vera verify; Tier-3 contracts trap at runtime. N/A On by default with vowc build; --no-verify to skip.
Repair loop Implicit: Pilot drafts predicates, prover checks. Implicit: write → verify → JSON error → fix. fix --plan --json proposes; agent applies. Explicit CEGIS, marketed as the workflow.
Honesty about uncertainty Per-VC verdicts include "could not prove". Silent Tier-3 fallback counted in tier3_runtime. No claim being made. Bounded; bound is stated; failure is explicit.

Z3/Why3 and ESBMC are complementary, not strictly ordered. SMT-deductive systems are better at pure arithmetic and quantified contracts over abstract values. BMC is better at concrete bug-finding in imperative loops and memory access patterns. Vow gets witnesses; MoonBit/Vera get quantifiers. Witnesses are what the agent loop actually consumes — "here's the input that breaks it" is a stronger signal than "could not prove goal X" — which is why Vow's bet is well-positioned even though Why3+SMT is a more general formalism.

5 · Skill packaging compared

How each project hands its language to the agent — an underrated UX axis. Four very different choices:

ProjectAlways-loaded bodyWhere the rest livesVersion-pinning
MoonBit No SKILL.md Pilot is the agent layer; no Claude Code skill, no Cursor profile, no system prompt. Pilot reads docs itself; closed loop. By tool, not skill.
Vera 99 KB / ~25 K tokens single file. All inline. Stdlib (7.8 K tokens) + Effects (3.2 K tokens) make up ~45% of the file. By file content; manual sync.
Zero ~1.5 KB discovery stub. Inside the compiler binary. Served via zero skills list / zero skills get <id> [--full]. Seven micro-skills, ~17 KB total. Binary-pinned: skills move with compiler version; zero drift.
Vow 1.7 KB / 48-line entrypoint. Linked reference files (grammar.md, cli.md, contracts.md, errors.md, examples.md) on disk; full self-contained bundle (~30 K tokens) printed on demand via vowc skill print --bundle. Generated from compiler; check_help_coverage.py catches drift.

Vera's monolith is the comparison that proves Vow's progressive-disclosure choice. At ~5× the practical ceiling for an always-loaded Claude Code skill, Vera is making a bet that the agent won't follow links — defensible only for agents that don't.

Zero's binary-pinned twist is more interesting. Treating the skill as something the compiler emits at runtime closes the version-skew window completely. Vow has the affordance (vowc skill print) but ships the full content on disk; going further would mean turning the on-disk stub into a discovery pointer that says "ask the binary".

6 · Diagnostics & repair

The diagnostic JSON shape is converging. All four projects ship structured per-error objects with codes and severities. The variation is in what extra fields the agent gets:

Vow has the unique witness + blame story. The missing pieces (italicised in the warning tags) are all portable from Vera and Zero with no language-level changes — documentation, plus an enrichment of the JSON emitter.

7 · Consolidated lessons for Vow

Deduplicated across all three comparisons, in rough priority order. Source attribution: M MoonBit V Vera Z Zero.

  1. Inference as a tracked effect. V
    Vera puts LLM calls in the effect row: Inference.complete : String -> Result<String, String>, effects(<Inference>). A pure function can't accidentally call out to a remote model — it's a type error. If Vow ever grows agent-callable builtins (LLM-as-stdlib), this is the right shape. Cost: one effect kind, one runtime shim. Zero verifier impact. The single strongest portable idea across the three comparisons.
  2. fixSafety ladder on every diagnostic. Z
    Vow has Blame (who is at fault) but no notion of how dangerous a fix is. A 5-level ladder — format-only, behavior-preserving, local-edit, api-changing, requires-human-review — lets an agent triage automatically. Vow mapping: strengthening a requires → api-changing; adding a loop invariant → local-edit; a counterexample showing a logic bug → requires-human-review. Complements blame.
  3. Typed repair handle alongside the diagnostic. Z
    Today a vow violation says "requires: y != 0 failed with y == 0". Adding repair: { id: "guard-divisor", summary: "Add precondition or rescue" } turns the diagnostic into an actionable JSON object the agent can dispatch on. The compiler doesn't have to write the patch — just name it.
  4. Fix example per error code. V
    Vera's errors.md ships a 4–6 line wrong/right snippet for every E###. Vow's errors.md has rationale and description but rarely a concrete fix shape. Documentation-only change; tightens the diagnostics-as-instructions story.
  5. Spec_ref anchors in diagnostic JSON. V
    Each Vera error carries spec_ref: "spec/05-contracts.md#postconditions". Vow's spec is already cleanly chaptered (docs/spec/{grammar,cli,contracts,errors,examples}.md) — embedding the anchor in the diagnostic JSON closes the loop. The agent doesn't have to guess where to read.
  6. Contract-driven property testing. V
    vera test --trials N uses Z3 to synthesize inputs satisfying requires, runs the function, asserts ensures. Vow already has vowc test; adding a --from-contracts --trials N mode reuses CEGIS infrastructure to generate random witnesses and exercise ensures at runtime. Belt-and-braces beside ESBMC; useful where BMC bounds are tight.
  7. Two-phase fix: vowc fix --plan --json. Z
    zero fix --plan --json proposes; the agent applies. Vow could emit a structured proposal — not a patch — covering candidate strengthenings of contracts, loop invariants, or guards. The plan can be regression-tested independently of any patcher.
  8. vowc explain <error_code> --json. Z
    Today VowViolation taxonomy / blame model / link to contracts.md live only in markdown. Lifting them into a runtime command makes them discoverable from the JSON itself: an agent that sees a code in build output can ask the binary what it means.
  9. vowc skill list / vowc skill get <id>. Z
    Zero ships seven micro-skills (~2–3 KB each) the agent fetches on demand. Vow's reference files are already structured this way (grammar, cli, contracts, errors, examples) — the missing piece is a discoverable index command. Closes the version-skew window further.
  10. Centralized "Do nots" page in the skill. Z
    Zero's zero-agent opens with six imperatives. Vow's equivalents are spread across contracts.md and CLAUDE.md. A short, top-of-skill list. Vow-specific candidates: don't weaken contracts to appease ESBMC; don't add n <= 10-style verifier bounds to requires; don't use --no-verify to skip a real failure; don't invent CLI fields or syntax.
  11. Bounded quantifiers in contracts. V M
    Vera's forall(@Nat, length, fn(...) {...}) and MoonBit's ∀ idx : Int, 0 <= idx && idx < h+1 → ... are quantifiers over known-bounded ranges. Translates straightforwardly to ESBMC as an unrolled loop of assertions. A concrete win for contracts that today require N copy-pasted ensures lines (e.g. "every element of xs[0..n] is positive"). Stays within the verifier-friendliness gate of Vow's "crisp rule".
  12. Inline test "name" { ... } blocks. Z M
    Both Zero and MoonBit have inline tests in any source file. Vow has tests/run/*.vow directory tests; inline tests would be a complementary mode for unit-level coverage. Bonus from Zero: xfail:-prefixed test names that surface unexpectedPasses in JSON — an agent sees instantly when it has accidentally fixed something.
  13. Inline snapshot tests with auto-rewrite. M
    MoonBit's inspect({ x: 10 }, content="{ x: 10 }") rewrites the content= string in source on --update. No separate golden directory. Tight loop for golden-output tests. Worth borrowing for Vow's tests/run/.
  14. Surface the cache key / fingerprint in JSON build output. Z
    Zero exposes cacheKeyInputs, snapshotKey, interfaceFingerprints. Vow already has a deterministic compile-object cache (see cli.md); making the key visible gives agents a stable handle for cache diagnostics, regression detection, and reproducibility audits. Builds on the binary-fixed-point story.
  15. Mandatory decreases on pure recursion. V M
    Both Vera and MoonBit require an explicit termination measure on recursion. Vow uses bounded unwinding instead. Borrowing the idea — an explicit termination measure the agent has to state — might let Vow drop the unwind bound for some cases or sharpen counterexamples. Feasibility note in the contracts ADR, not an immediate change.
  16. Forward-looking: constrained decoding is the layer above Vow. M
    MoonBit attacks "agents emit invalid code" at the sampler; Vow attacks it at the verifier. These compose: a Vow-aware sampler could rule out unparseable or trivially ill-typed tokens before ESBMC ever runs. Worth a forward-looking note in docs/spec/. Not an action item.
  17. One canonical form, optionally enforced. V
    Vera ships vera fmt --check as part of the workflow; there is no style debate. Vow already has a canonical printer (a compiler pass), but doesn't enforce it as part of vowc build. Optional CI gate.

8 · What Vow should not borrow

Each project's identity is built on choices that fight Vow's identity. Picking these up would dilute, not strengthen, what Vow is for:

From Vera

  • De Bruijn slot references. Vera's defining bet, and the most contested by every external reviewer. Bench wins are model-specific (Kimi K2.5 100%, Sonnet 4 79% — worse than Python). Names anchor LLM attention; grep matters; the cost to human collaborators is real. Thesis too narrow and too disputed to bet on.
  • Tier-3 runtime fallback for verification. If Z3 can't discharge a contract, Vera silently emits a runtime assert and counts it in tier3_runtime. Breaks the "verified by default" pitch. Adopt the counting in JSON; don't adopt the fallback.
  • WASM-only backend. Cranelift native + binary fixed-point self-host is one of Vow's strongest credibility signals. Vera's Python-host + WASM-only target is research-grade.
  • Mandatory effect/visibility on every function. Vera requires public/private, requires, ensures, effects on every declaration. Vow's "pure-by-default, effect annotations only when needed" carries the same information with less syntactic load.
  • No loops / no mutation / no linearity. Vera's "everything is recursion, everything copies" gives up Vow's linear-types story. Linear types are load-bearing for Vow's memory-discipline brand. Don't trade them.
  • Async-as-a-marker-only. Vera's Async effect has "no runtime overhead" because it's sequential. Same syntax, different semantics — an agent has no way to tell from the call site. If Vow ever adds async, it should be real.
  • Monolithic 99 KB SKILL.md. Wrong default for Claude Code specifically. Vow already does the right thing.

From Zero

  • Generics + meta(...) compile-time introspection. Zero has both. Vow has deliberately rejected both because they expand the verification surface. Generics force monomorphization complexity that ESBMC has to chase. The "crisp rule" in CLAUDE.md — "add surface sugar only when it desugars to today's core semantics with near-zero verifier impact; reject anything that introduces a new type-system axis" — argues against this directly.
  • Sub-10 KiB binary chase. Vow's binaries can be larger if it buys verification. Don't optimise away ESBMC's static C model for binary-size points.
  • Removing the C backend. Zero removed its generated-C path in favour of direct emitters. Vow needs the C backend for ESBMC. Different priorities.
  • 25 subcommands. A lot of Zero's surface (zero mem, zero ship, zero abi, zero doctor) is useful only at productized-toolchain scale. Vow should pick subcommands by clear agent need, not parity.
  • Effect-as-capability syntax. Threading world: World through every effectful function is grep-friendly but visually heavy. Vow's effect sets on function types do the same job structurally with less noise. Adopt the explicitness; keep the syntax.

From MoonBit

  • Proprietary protocols (ASP). Closed, undocumented, ships with the compiler. Vow's bet on "JSON-everywhere, no proprietary protocol" is the better story if the agent ecosystem is going to be Cambrian.
  • Source-available license. MoonBit's MPSL (SSPL-style anti-cloud clause) is a recurring friction point in HN discussion. Vow's open-source posture is both straightforward marketing and engineering.
  • Vertically integrated agent (Pilot). Pilot is a closed loop: own sampler, own protocol, own agent, own UI. Even if technically excellent, it's a separate product for the language team to maintain. Vow's "JSON-everywhere, any agent" is more leverage-efficient.
  • Separate predicate files (.mbtp). Tempting for reuse and a richer logical sublanguage (quantifiers, recursive predicates) — but a new namespace, new file type, new logic. Fails the "doesn't make verification harder" gate unless there's a concrete reuse story Vow can't currently express. Inline contracts are simpler and currently sufficient.
  • Why3 as the verification primitive. Real escape hatch if Vow ever needs unbounded ∀/∃, but adopting it means giving up concrete counterexamples. The CEGIS loop changes shape from "here's the failing input" to "could not prove goal X". Not worth the trade.

9 · Where Vow's moat is

After three comparisons, the picture is clearest about what Vow has that no one else does:

AssetHeld byDefensibility
Concrete counterexamples (failing input + execution trace + values) Vow alone Architectural — tied to BMC. MoonBit (Why3) and Vera (Z3) can't get there without a different backend. Strong moat.
Caller/Callee blame on every violation Vow alone Cheap to copy; but no one else has thought to. Defensible by being first and getting it right.
Linear types + no GC Vow alone (in this set) MoonBit is GC; Vera has no memory story; Zero has owned/ref but not linearity. Linear types are load-bearing for Vow's brand. Hard to retrofit elsewhere.
Binary fixed-point self-host Vow alone (in this set) MoonBit isn't self-hosted; Vera isn't; Zero's self-host is in progress. Credibility signal that scales with time invested.
Verification on by default Vow alone MoonBit requires "proof-enabled": true per package; Vera's verify is a separate verb; Zero has no verifier. Vow's "vowc build verifies unless you opt out" is uniquely strong.
Small core surface (no generics, no traits, no closures, no macros) Vow alone MoonBit, Vera, Zero all ship generics. Vow's "crisp rule" makes the verification surface tractable. Reverse-defensible: every feature Vow keeps out of the language is one ESBMC doesn't have to chase.
Published cross-language benchmarks (36/36, HumanEval 66/67, 103/103 references) Vow leads Vera has VeraBench (single-run, self-acknowledged unreliable). MoonBit has the LLM4Code 2024 paper but no equivalent public agent bench. Zero has none.

None of the three threaten any of these. Where the others do beat Vow — fixSafety ladders, typed repair handles, spec_ref anchors, the Inference effect — the gaps are either documentation-only or JSON-shape changes. They don't require language work or verifier changes.

Bottom line

In three months the agent-language landscape went from "is this a thing?" to four credible projects all converging on the same diagnosis. The right read for Vow is:

  1. The category is real. Stop hedging on positioning. "AI-friendly language with provable contracts and concrete counterexamples" is now a contested space; lean into it.
  2. Vow's moat is structural (BMC counterexamples, blame, linear types, self-host, verification-on-by-default), not cosmetic. None of MoonBit, Vera, or Zero threatens it.
  3. Most of what's worth borrowing is documentation or JSON-shape work. Five high-priority items above could ship in a single release without touching the verifier, the type system, or the compiler architecture.
  4. The two language-level lessons that might be worth taking later — bounded quantifiers and Inference as a tracked effect — both clear the "crisp rule" gate (no new type-system axis, no verifier impact). Hold them in reserve for when there's a concrete pull.
  5. Reject loudly the choices that look like wins but would dilute Vow: De Bruijn names, Tier-3 runtime fallback, generics+metaprogramming, monolithic SKILL.md, Pilot-style proprietary protocols. These are the others' identity, not Vow's.
