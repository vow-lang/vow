Vera SKILL.md — analysis & lessons for Vow

A structural look at the skill document veralang.dev/SKILL.md ships, with a side-by-side comparison to how Vow ships its own Claude Code skill.

Vera SKILL.md snapshot fetched 2026-05-18 · Vow entrypoint + bundle built from scripts/generate_help.py at vow/compareVera.

TL;DR

  • Vera ships a single 99 KB / ~25 K-token SKILL.md. That is roughly 5× the rough ceiling Anthropic suggests for a SKILL.md.
  • Two sections — ## Built-in Functions (32 KB) and ## Effects (13 KB) — make up ~45 % of the file. They are reference material that would normally live behind a link.
  • Vow already does the thing Vera should do: a 1.7 KB / 48-line entrypoint SKILL.md with reference links plus a separate ~30 K-token bundle for raw-API contexts.
  • The right takeaway from Vera is not "shrink the skill". It is (a) Vera proves the stdlib-as-skill-content idea has a non-zero cost we should name, and (b) there are a handful of content ideas worth borrowing (Inference effect, tiered verification, fix-example-per-error-code) — independent of the skill packaging.

4. Is Vera's SKILL.md really “too large”?

Two angles:

Angle A — packaging-as-a-Claude-Code-skill

Yes. A Claude Code skill is loaded on trigger and consumes context for the remainder of the session. At ~25 K tokens that is a significant slice of any context window before the user has typed anything. Anthropic's own skill examples target much smaller bodies with linked sub-resources. By that bar, Vera's SKILL.md is over by roughly 5×.

Angle B — packaging-as-a-self-contained-language-reference

Less obvious. Vera's design ethos is “everything an agent needs in one fetch”. The benchmark wins reported in vera-bench may partly rely on the full surface area of the stdlib being in context. Splitting it into linked docs has a real cost if the agent doesn't follow links — and many agents in production don't.

So the more careful claim: Vera is making an explicit bet that fetching the whole surface beats progressive disclosure. That's a defensible bet for a language whose entire pitch is “designed for LLMs to write”. The skill being “too large” is true only if you accept the framing that a skill's job is to point at material, not to be the material.

5. What Vow can learn — concretely

Two categories: skill packaging (where Vow is already ahead) and content / language design (where Vera has a few ideas worth borrowing).

Skill packaging — what we already do right, but should formalize

  • Entrypoint + linked references is the right shape. Vera's monolith is the comparison that proves it. Keep doing this.
  • Make the bundle discoverable. Right now vowc skill print --bundle exists but is undocumented in the entrypoint. Add a one-line "for raw-API use, fetch the bundle" note. Cost: trivial.
  • Audit the bundle for sections that should be lazy. The bundle is ~30 K tokens. If anyone embeds it in a system prompt (the bench harness does), the same bloat critique applies. Candidates to elide from bundle: full error catalog (336 lines of ## Compile-Time Errors), full JSON schemas.
  • Keep when_to_use separate from description. Vera collapsing them is a packaging regression. Stay split.

Content / language design — ideas worth stealing

IdeaFromWorth it?Cost
Inference as a tracked effect. Calling an LLM is part of the effect row, so a pure function cannot accidentally call out. Vera <Inference> effect with Inference.complete : String -> Result<String, String> Yes if Vow grows agent-callable builtins Effect kind + runtime shim. No verifier impact.
Tier-3 runtime fallback. If an ensures can't discharge statically, emit a runtime assert and label the function "verified-with-runtime-fallback" in build output. Vera's three-tier strategy Possibly — could weaken Vow's "verified by default" pitch. Code emission + JSON status enum. Risk: distorts the brand.
Fix-example per error code. Each E### ships a wrong/right snippet inline. Vera's stable codes E001–E702 with examples Yes Documentation only. Vow's errors.md mostly has rationale, less often a fix example.
Contract-driven property tests. vera test --trials N auto-generates inputs satisfying requires, runs them, asserts ensures. Vera toolchain Yes — Vow has vowc test already; could grow a contract-driven mode. Random input generator + runtime assert in debug mode. Reuses existing infrastructure.
Typed holes (?) producing a warning, not an error. Lets the agent type-check incrementally. Vera W001 Maybe Parser + checker support. Useful but small.

Ideas to not steal

  • De Bruijn slot references. Vera's defining feature, and the most contested. HN consensus was sceptical; cross-model bench numbers were mixed (Kimi K2.5 100 %, Claude Sonnet 4 79 %). Names anchor attention; grep matters; the cost to human collaborators is real even in an agent-first language.
  • Effects without linear types. Vera has no mutability, no linearity, no ownership story — everything copies. Vow's linear types are differentiated and worth keeping.
  • WASM-only backend. Vow's Cranelift native pipeline + binary fixed point is a credibility signal that Vera doesn't have.

6. What this means for Vow's own skill

Vow's entrypoint is already small. Two small improvements would still be worth making:

  1. Surface the bundle path in the entrypoint. One line: "For self-contained reference (e.g. raw API use), run vowc skill print --bundle." Helps anyone wiring Vow into a non-Claude-Code agent harness.
  2. Add a fix-example field to errors.md. Each E### already has a description; adding a 4–6 line wrong / right pair brings parity with Vera's diagnostics-as-instructions framing. The agent doesn't need to guess the fix shape.

Both are low-cost and don't require changes to the language or the verifier.

7. Bottom line

Vera's SKILL.md is too large as a Claude Code skill — about 5× over the practical ceiling — and the bloat is concentrated in two reference-material sections (stdlib + effect APIs) that progressive disclosure handles cleanly. Vow already does the right thing by shipping a 1.7 KB entrypoint plus an opt-in bundle.

The interesting lessons are not in the packaging, where Vera is just paying for a different choice. They are in three small content moves Vera makes that Vow could pick up for very low cost: Inference as an effect, fix examples per error code, and contract-driven test generation. Skip the De Bruijn names, the WASM-only target, and the runtime-fallback tier — those are Vera's identity, not Vow's.
